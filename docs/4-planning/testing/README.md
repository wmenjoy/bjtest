## 📄 子目录说明

### active/ - 活跃测试计划

**用途**: 正在执行或即将执行的测试计划

**生命周期**:
- 创建时机：版本开发启动或测试阶段开始
- 更新频率：频繁（每周或每天更新进度）
- 移出时机：测试完成后移至 `completed/`

**命名规范**:
- 版本测试计划：`v{version}-test-plan.md` (如 `v2.0-test-plan.md`)
- 专项测试计划：`{type}-test-plan.md` (如 `regression-test-plan.md`, `security-test-plan.md`)

**示例文档**:
- `v2.0-test-plan.md` - v2.0版本完整测试计划
- `regression-test-plan.md` - 回归测试计划（持续执行）
- `integration-test-plan.md` - 集成测试计划

### backlog/ - 待办测试计划

**用途**: 计划但尚未开始的测试活动

**生命周期**:
- 创建时机：识别测试需求但暂不执行
- 更新频率：低（优先级调整时更新）
- 移出时机：开始执行时移至 `active/`

**命名规范**: 与 active/ 相同

**示例文档**:
- `performance-test-plan.md` - 性能测试计划（计划Q2执行）
- `load-test-plan.md` - 负载测试计划
- `security-audit-plan.md` - 安全审计计划

### completed/ - 已完成测试计划

**用途**: 已完成的测试计划归档（按季度）

**目录结构**: `YYYY-QN/` (如 `2024-Q4/`, `2025-Q1/`)

**生命周期**:
- 创建时机：测试计划完成
- 更新频率：无（仅供历史参考）
- 归档规则：按完成时间的季度归档

**价值**:
- 历史回顾：了解过往测试覆盖范围
- 经验积累：参考历史测试计划改进流程
- 审计追溯：质量审计时查阅

## 🔄 测试计划生命周期

```
1. 创建计划
   ↓
2. 放入 backlog/ (如暂不执行)
   或 active/ (立即执行)
   ↓
3. 执行测试 (在 active/ 中频繁更新进度)
   ↓
4. 测试完成 (标记状态为 ✅ 已完成)
   ↓
5. 归档到 completed/YYYY-QN/
```

---

## 📊 测试计划状态管理

**状态标记**:
- 🔄 进行中 - 测试正在执行
- ⏸️ 暂停 - 因外部原因暂停
- ✅ 已完成 - 测试已完成，可归档
- ❌ 已取消 - 计划取消（说明原因）

**更新原则**:
1. **每日更新**: 在 active/ 中的计划，每日或每周更新进度
2. **完成即归档**: 测试完成后，立即移至 completed/
3. **记录变更**: 所有重要变更记录在文档末尾

## 📌 最佳实践

1. **提前规划**: 版本开发初期就创建测试计划
2. **细化任务**: 将测试计划拆分为可追踪的任务
3. **实时更新**: 测试进度变化时及时更新文档
4. **关联需求**: 测试计划必须关联需求文档
5. **风险前置**: 提前识别风险并制定缓解措施
6. **经验总结**: 测试完成后及时总结经验教训

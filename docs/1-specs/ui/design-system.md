## 设计系统概述

本目录包含测试平台的**技术性UI设计规范**，用于指导前端开发实现。

### 适用范围

本规范适用于需要明确技术实现细节的设计文档，包括：
- 组件技术规范
- 接口定义
- 状态管理设计
- 数据流设计
- 交互行为规范

### 与产品设计的区别

| 文档类型 | 存放位置 | 内容特点 |
|---------|---------|----------|
| **技术性UI规范** | `docs/1-specs/ui/` | 面向开发，包含技术实现细节 |
| **产品设计文档** | `docs/2-requirements/designs/` | 面向产品，包含需求和线框图 |

## 设计规范分类

### 1. 组件设计规范 (components/)

定义可复用UI组件的技术实现规范。

**文档模板**：
```markdown
# 组件名称

## 技术规范
- React组件签名
- Props接口定义
- 状态管理方式

## 样式规范
- CSS类名约定
- 主题变量

## 交互行为
- 事件处理
- 状态变化

## 数据流
- Props数据流向
- 事件回调

## 示例代码
```

### 2. 布局设计 (layouts/)

定义页面布局系统和响应式规则。

**包含内容**：
- 网格系统（12列、24列等）
- 断点定义（mobile, tablet, desktop）
- 页面容器规范
- 间距系统（margin, padding）

### 3. 交互设计规范 (interactions/)

定义用户交互的技术实现细节。

**包含内容**：
- 动画时长和缓动函数
- 状态转换规则
- 拖拽API使用
- 键盘快捷键映射

## 使用指南

### 创建新的设计规范文档

1. **确定文档类型**：
   - 技术实现细节 → 放在 `docs/1-specs/ui/`
   - 产品需求设计 → 放在 `docs/2-requirements/designs/`

2. **选择子目录**：
   - 组件规范 → `components/`
   - 布局规范 → `layouts/`
   - 交互规范 → `interactions/`

3. **命名规范**：
   - 使用kebab-case：`component-name.md`
   - 简洁明确：`button-group.md` 而非 `button-group-component-spec.md`

4. **文档结构**：
   - 包含技术接口定义
   - 提供示例代码
   - 说明状态管理方式
   - 注明依赖关系

### 文档评审流程

1. 技术评审：确保规范可实现
2. 设计评审：确保符合设计系统
3. 开发评审：确保规范清晰易懂
